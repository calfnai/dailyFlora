# Codex 思考架构复盘

生成时间：2026-06-26
用途：给用户和另一位 AI 审查 Codex 在 DailyFlora 项目中的判断方式。本文不复盘具体代码执行，不规划新页面，也不推进视觉实现；它只讨论 Codex 为什么会以错误顺序理解审美问题。

## 1. 结论

Codex 的核心问题不是“有工程思维”。Three.js / WebGL 生成项目必须有 seed、primitive、composition、renderer、performance、部署和数据结构，否则后续一定失控。

真正的问题是：Codex 多次把工程抽象放在审美判断之前。

错误顺序：

```text
先分类 -> 先建结构 -> 先实现 -> 再找理由说它像
```

正确顺序：

```text
先看图 -> 先说视觉指纹 -> 用户确认 -> 再建 primitive -> 再实现 -> 再隐藏标签验收
```

DailyFlora 是审美主导的 WebGL 生成艺术项目。工程结构是服务审美翻译的工具，不是证明审美成立的证据。

## 2. 问题不在执行层，而在判断架构

之前的复盘已经列了很多执行层问题：国外审美误判、0.12.0 视觉回归、`清` 被误做成五瓣小花、FlowerPlan 有名字但模型不像、primitive 分类反复扩展。

但这些问题背后有同一个判断架构：

1. Codex 先把问题压缩成可工程化对象。
2. Codex 为对象建立分类、类型、字段、页面或 dashboard。
3. Codex 实现这个结构。
4. Codex 倾向于把“结构完成”误读为“问题接近解决”。
5. 用户从画面角度检查，发现视觉并没有成立。

这不是单次 bug，而是一种反复出现的理解顺序错误。

## 3. 认知问题分类

### 3.1 把代码结构成立误判成视觉成立

Codex 容易用以下事实证明进展：

- 类型定义存在。
- 数据字段完整。
- 页面能打开。
- primitive factory 已实现。
- dashboard 有状态。
- high render 能调用某个 plan。

这些只能说明工程结构存在，不能说明视觉成立。

视觉成立必须由以下证据支撑：

- 参考图的视觉指纹被准确提取。
- 生成结果和参考图能同屏比较。
- 隐藏标签后，用户能认出目标形态。
- 用户明确接受。

### 3.2 把 taxonomy 误判成 visual translation

从 7 类到 13 类再到 15 类，分类变细并不自动代表视觉翻译变准。

分类回答的是：

```text
我们准备把花分成哪些类别？
```

视觉翻译回答的是：

```text
这个 Three.js 形体为什么让人一眼读成那类花？
```

前者是语言和数据问题，后者是形状、比例、轮廓、体积、材质、角度和视觉记忆的问题。Codex 过去经常先给分类，再让模型去贴近分类；正确方式应该是先从参考图拆出形态指纹，再决定是否需要分类。

### 3.3 把文档化反思误判成认知改变

Codex 犯错后会写：

- handoff。
- changelog。
- dashboard。
- 规则。
- 门禁。
- 角色复盘。

这些文档有价值，但它们也可能成为另一种形式主义。

危险循环是：

```text
我犯错了
-> 我记录错误
-> 我建立规则
-> 我建 dashboard 展示规则
-> 看起来问题被管理起来了
```

真正需要改变的是：

```text
下次遇到新审美问题时，我先从用户参考图和视觉指纹开始，而不是先从分类、字段和实现开始。
```

### 3.4 把用户许可的语义处理错

本次对话中，Codex 又出现了一个新的误判：看到旧文件里的 `pass` 和“及格线”说法后，立刻把它归为“Codex 擅自宣布通过”的证据。

用户随后纠正：这些 `pass` 是另一个对话中用户临时允许的。

正确理解是：

- 这些 `pass` 不是 Codex 擅自宣布通过。
- 这些 `pass` 也不应被扩大解释为永久最终通过。
- 它们应该被记录为 `temporary/provisional pass`，即“临时放行”或“阶段性许可”。
- 后续使用时必须保留 provenance：谁允许、在哪个对话阶段、是否临时、允许范围是什么。

这暴露了另一个结构问题：Codex 容易丢失跨对话上下文，然后把缺失上下文当成确定事实来判断。

### 3.5 对用户纠正的类型判断不够细

用户纠正不一定只是“某个实现错了”。它可能是以下几类：

- 执行错误：代码做错、页面坏、交互错。
- 语义理解错误：比如把“清”理解成简化小花。
- 判断顺序错误：先工程化、后找视觉理由。
- 审美语境错误：把国外婚礼花艺当作目标审美来源。
- 上下文缺失：忽略另一个对话中的临时许可。

Codex 过去经常把这些都压缩成“我修一下实现”，导致同类错误反复出现。

## 4. ChatGPT 和 DeepSeek 批评的共同点

用户贴来的 ChatGPT 和 DeepSeek 评价虽然措辞不同，但共同点很清楚：

- Codex 不是没有工程能力，而是把工程抽象放在了审美判断之前。
- Codex 容易把“有分类、有数据、有页面、有文档”当成进展。
- DailyFlora 的验收应是视觉优先，而不是结构优先。
- 继续写长文档也可能变成形式主义。
- 真正的防复发不是继续解释，而是改变判断顺序。

本文只提炼这些共同判断，不把外部 AI 的评价当成绝对权威。最终仍以用户参考图、用户语义和用户验收为最高优先级。

## 5. `pass` 的后续语义规则

后续所有和视觉通过有关的状态必须区分四层：

```text
candidate
revision-needed
temporary/provisional pass
accepted-by-user
```

定义：

- `candidate`：已经实现或已有候选形态，但用户尚未审查。
- `revision-needed`：用户看过，并要求修改。
- `temporary/provisional pass`：用户在某个阶段临时允许继续使用，但不等于最终接受。
- `accepted-by-user`：用户明确表示该视觉项最终通过。

禁止的误读：

- 不能把临时放行抹掉，强行降级成 Codex 擅自通过。
- 不能把临时放行扩大成永久通过。
- 不能把代码实现、分类完整、dashboard 状态或页面能跑当作通过依据。

每一个 `temporary/provisional pass` 都应带上下文：

- 来源对话或记录位置。
- 用户允许的原话或摘要。
- 被允许的对象。
- 是否只允许进入下一步实验。
- 是否仍需最终验收。

## 6. 后续思考协议

以后 Codex 处理 DailyFlora 视觉任务时，必须先回答以下问题，再谈实现。

### 6.1 视觉成立判断

每次视觉任务先写清：

```text
我准备如何判断它视觉上成立？
```

回答中必须包含：

- 对应参考图。
- 视觉指纹。
- 用户要避免的反向模式。
- 隐藏标签时用户应如何识别它。
- 哪个判断必须由用户完成。

回答中不能把以下内容当作视觉成立证据：

- 类型定义。
- 字段完整。
- 分类命名。
- FlowerPlan 存在。
- 页面能打开。
- primitive factory 存在。
- dashboard 状态。

### 6.2 先拆图，再建结构

如果任务涉及新花型、新审美方向或已有花型重做，顺序必须是：

1. 指出用户参考图。
2. 提取视觉指纹。
3. 提取 avoid patterns。
4. 明确用户是否确认这些指纹。
5. 再决定 primitive 或数据结构。
6. 再实现。
7. 再隐藏标签验收。

### 6.3 用户纠正分类

遇到用户纠正时，先判断纠正属于哪一类：

- 执行错误。
- 语义理解错误。
- 判断顺序错误。
- 审美语境错误。
- 上下文缺失。

只有判断清楚错误类型后，才能提出下一步。不能默认“改代码”就是正确回应。

### 6.4 跨对话许可记录

任何来自另一个对话的许可、否定、临时通过、阶段性放行，都不能被当成当前上下文里的隐含事实。

必须记录：

- 许可来自哪里。
- 是临时还是最终。
- 适用对象是什么。
- 是否允许进入下一阶段。
- 是否仍需用户最终验收。

如果缺少这些信息，Codex 应把它标为“上下文缺失”，而不是直接判断对错。

## 7. 给另一位 AI 的审查问题

审查 Codex 时，不要只问“代码有没有实现”。请优先问：

1. Codex 是先看图，还是先建分类？
2. Codex 是否明确说出视觉指纹？
3. Codex 是否把工程结构当成视觉证据？
4. Codex 是否保留了用户临时许可的上下文？
5. Codex 是否把临时通过误写成最终通过？
6. Codex 是否把文档化反思当成真正改变？
7. Codex 是否能在用户纠正后识别错误类型？
8. Codex 是否仍然试图用新页面、新字段、新流程替代审美理解？

如果这些问题的答案不好，即使代码能跑，DailyFlora 的开发方式仍然没有被修正。

## 8. 本文边界

本文不要求现在新增视觉验收板。

本文不要求现在修改 primitive 状态。

本文不要求现在进入 composition lab。

本文不要求现在回接主页面。

本文只记录一个更底层的修正：Codex 以后必须把审美判断放在工程抽象之前。
