# DailyFlora 0.13 Aesthetic System

最后更新：2026-07-13

这份文档是 DailyFlora 0.13 的审美记忆。它把本项目到目前为止由用户纠正出来的判断写进仓库，供后续 Codex、人工开发和审美复盘共同使用。

0.13 的重点不是新增更多花，而是把判断机制固定下来：DailyFlora 要生成什么样的花束、哪些方向已经被否定、下一次视觉改动前必须怎么读参考、怎么验收、怎么记录。

2026-07-03 的 `0.14.0` 在此基础上把当前 dashboard 中的主要审美组统一确认为通过基线：晨露莓园、夏日风车、梦幻紫、洋水仙季节、会呼吸的风景、狐尾百合、浆果森林、秋日果汁、荔枝花园彩虹和反向边界。后续不再把这些组当作待验收阻塞；如果实际运行页面暴露问题，按用户反馈新增修正记录。

## 读者

当前读者首先是用户本人。Codex 可以读取、执行和更新这份文档，但不能把自己的审美判断当成最终验收。

后续如果用户放权，放权也只发生在这个系统已经稳定执行之后，不是因为 Codex 说“我理解了”。

## 目标花束

DailyFlora 的目标不是通用“好看的花”，而是有中国当代花艺和小红书语境的数字花束：

- 整体有空气感，不是一坨密实花头。
- 花束有整体，也有局部细节；远看轮廓成立，近看花材有差异。
- 色彩可以高饱和，但需要叶材、绿色和空气距离来缓冲，避免油画式腻感。
- 小花矩阵、低位簇花、莓果点、杯形/蝶形/星形花和外开线条应共同形成节奏。
- 花束需要春意、湿润感、轻盈感和日常可观看性，不要端庄婚礼感或批发市场感。
- 360 度旋转时仍要成立，不能只在单一正面好看。

目前最有启发的正向样本是“山岗小花”：小花矩阵承接颜色，低位簇花托住体积，穗状花穿插而不成片，线条向外打开，绿色托住花与空气之间的空气。

“晨露莓园空气束”是 0.12 后期第一个比较接近用户预期的原创候选：它学习山岗小花的空间语法，但换成清晨莓园方向，靠奶白小花、蓝紫雾团、莓红点和嫩绿外开线营造湿润空气。

## 反向约束

下列方向默认不采纳：

- 欧美婚礼花艺、bridal bouquet、market bouquet、Pinterest 式国外花艺作为默认审美。
- 单一大主花统治画面。
- 粗糙市场束，花材像随机插上去。
- 全是花、没有绿色缓冲，导致画面过腻。
- 花头和绿色植物分层过度，破坏整体轮廓。
- 穗状花在顶部成片堆积，或全部朝同一方向、全部向内聚拢。
- 叶材挡住主花、飞到不合理位置、比花更抢戏。
- 为了技术复杂度增加更多几何，但视觉上离参考更远。
- 洋桔梗。用户明确表示个人特别讨厌它，因此它不进入默认花库、花束计划或玫瑰型参考。

## 叶材与绿色部分

叶材和绿色部分是空间感支撑，不是物理支架。

真实花束由花泥、扎口或内部结构固定；绿色部分不需要真的托住花盘。它的作用是：

- 带来层次感和春意。
- 缓冲高饱和花色，避免全花头的油腻感。
- 通过花和叶的不同朝向、不同距离，形成花束会呼吸的感觉。
- 从花束结构里合理冒出来，托住花间的空气。
- 拉开空间，补充边缘节奏。

叶材不能：

- 挡住主花。
- 随机飞到不合理位置。
- 在空旷位置呈现根部草束那种上宽下窄的孤立形态。
- 比花更抢戏。

## 花库记忆

Primitive Lab 的旧 7 类只是历史粗分类，不足以覆盖主流花束形态。0.13 继承 16 类花库，用户已确认达到及格线：

1. 盘状花 / `disk-face-flower`
2. 波斯菊/小面花型 / `cosmos-open-face`
3. 层叠大丽花/团瓣型 / `layered-dahlia-form`
4. 褶皱玫瑰型 / `ruffled-rose-form`
5. 星形/风车型 / `star-pinwheel-form`
6. 郁金香/杯型 / `tulip-cup-form`
7. 洋水仙管心型 / `trumpet-throat-form`
8. 大喇叭型 / `datura-trumpet-form`
9. 兰花/蝴蝶型 / `orchid-butterfly-form`
10. 马蹄莲/卷曲苞片型 / `calla-curled-bract`
11. 穗状花 / `spike-vertical-form`
12. 伞状/小簇型 / `umbel-mini-cluster`
13. 绣球/云团型 / `hydrangea-cloud-cluster`
14. 果材/荚果型 / `fruit-pod-form`
15. 吊坠风铃果型 / `hanging-bell-fruit`
16. 叶材/草线/枝条型 / `foliage-grass-branch`

新增 `cosmos-open-face` 是因为用户指出山岗小花和热带丛林里有一种好看的小面花并未进入 TARGET SHAPE VOCABULARY。它比盘状花更轻薄、更少瓣、更适合承担小花矩阵和呼吸点，不能继续藏在 `chamomile` 或 `DiskFlower` 下面。

“达到及格线”不是最终完美，只表示可以进入整束 composition 验证。后续如果某一类在主花束里暴露新问题，要回到 Primitive Lab 或 dashboard 记录，而不是在整束里偷偷糊过去。

Primitive Lab 的展示本身也是花库门禁的一部分：正式花型和候选花型放在一个共享的大型 3D 花库画布里，每朵花占据清楚的位置，旁边直接显示中文名、自然英文名、识别说明和状态。不要为每一类建立独立 WebGL 渲染框；新增花型继续进入同一画布。实验工具负责切换视角、轮廓、网格和单花旋转，但不能取代词表说明。

2026-07-11 新增待验收候选：`FrilledNarcissusFlower / 褶边副冠水仙型`。它继承 trumpet-throat v1/v3 的六片花被、明显副冠、褶边、深喉、花蕊和绿色连接点，并转换成连续的 3D 结构。它暂不计入正式 16 类；只有用户确认其正面、侧面和 360° 查看都真正成立，才升级为第 17 类。

2026-07-11 新增独立的“偏写实花型 LAB”：它不改变 Primitive Lab 的 16 类抽象词表，而是把花名进一步拆成 25 种可直接识别的具象表达。名单来自用户给出的 26 种花材并明确删除洋桔梗。页面继续使用单一共享 3D 画布；每种花旁显示中英文名、形态说明和实体连接说明。

偏写实花型同时为潜在 3D 打印产品做结构准备：花瓣必须嵌入花心或花托，花头必须连到花梗，簇花必须有可见实体分枝，不允许漂浮零件。当前阶段只承诺结构连续和具有物理支撑；正式售卖前还必须完成布尔合并、最小壁厚、水密/流形检查、支撑与打印方向测试，不能把网页里的视觉模型直接当成可出厂 STL。

2026-07-12 起，偏写实 LAB 中的 16 种具体单花同时进入 Primitive Lab，作为抽象词表之外的具象扩展：雏菊、洋甘菊、非洲菊、太阳花、银莲花、波斯菊、大丽花、玫瑰、花毛茛、山茶、牡丹、乒乓菊、郁金香、洋水仙、蝴蝶兰、马蹄莲。这样生成器可以在抽象角色之外直接选择具体花材。飞燕草、金鱼草、风信子、狐尾百合、蛇鞭菊、蕾丝花、绣球、满天星、米花暂缓并入；它们当前形态需要后续与用户单独讨论修改。

2026-07-12 起增加第三层“科幻花型”方向：它不追求现实花材识别，而是在花瓣、花心、花蕊、花托和生长连接仍然成立的前提下，使用能量核、轨道、脉冲、非自然对称与发光色创造不存在的生命体。第一例是 `OrbitalPulseFlower / 星环脉冲花型`。科幻不能成为随意漂浮的几何装饰；所有环、节点和花蕊仍需连接花心，并在正面、侧面和 360° 旋转中读成花。

用户随后纠正：星环脉冲花的形态仍然写实，只是使用了科幻配色。2026-07-13 用户确认该花型可以验收，因此它从候选升级为正式可用的“写实骨架 × 非现实配色”混合花型；这项通过不等于结构科幻形态通过。真正的科幻花型必须在灰模下仍表现出非现实生长逻辑。配色必须成为 UI 上可独立改变的变量，不能再与形态写死。

## Composition 规则

整束花的审美优先级高于单个 primitive 的技术炫技。

目前应优先继承这些 composition 规则：

- 小花矩阵比大花单点统治更接近目标。
- 低位簇花负责托住体积，但不能变成厚重底座。
- 穗状花应穿插在花丛中，而不是集中在顶部。
- 穗状花单体可以直，但整束朝向要有外开、侧穿和随机差异。
- 线条应打开空间，不要全部向中心聚拢。
- 果材是局部点睛和童话感来源，不是大球堆。
- 花束默认镜头要能看清整体，再允许用户 zoom in 看细节。
- 360 度旋转时，花材面向、叶材方向和空隙关系都要合理。
- 每一枝穗状花的轴线必须能回溯到同一个真实扎口或花瓶开口，并有连续可见的花茎连接到花穗底部；不能只在半空摆放花穗本体，再用统一角度制造虚假的外开感。

2026-07-12 日期修正门禁：`2026-07-07` 的默认日期花束不得再回到中心团花、簇花密铺成球的状态，应以小花、果点、枝线和空气间隔形成星星点点的缤纷节奏；`2026-07-11` 的默认日期花束应缩短并下沉穗花，减少重复线花，并让每枝穗花服从共同花瓶扎口。两束花在用户复验前保持待验收。

每日可辨认性也是发布门禁：HUD 主标题不能只重复有限数量的主题名，必须显示日期印记和日期 seed 决定的独立中英文花名；主题、flowerPlan 和花材清单作为第二层信息保留。即使用户选择“疏/省”，也应能从标题直接确认这是新一天的花。

2026-07-13 新增每日花材丰富度门禁：用户指出当日默认花束不好看，主要原因是用到的花的品种不够丰富。排查后确认，`2026-07-13` 默认 seed 抽到 `tropical-forest / 热带丛林` 与 `berry-grove / 浆果森林束`，该 plan 仍主要使用老 `FlowerTypeId` 中的少数抽象角色，最终只映射到约 6 种 primitive。0.14.6 虽已把 16 种具象单花加入 Primitive Lab，但每日默认 flowerPlan 和主花束 renderer 尚未真正把这些具象单花纳入候选、配比和可见映射。因此后续每日生成不能只依赖主题换色、果点或旧抽象类型重复，必须建立最小花材丰富度，并让具象花库真正进入每日花束。

同日修正每日中文名重复 bug：用户指出 7 月 8 日和 7 月 9 日一类相邻日期可能出现相同中文名。原因是旧逻辑只用少量 mood 加 flowerPlan 名组合，日期印记虽然不同，但花花自己的中文名仍可能重复。后续默认命名允许凭审美起名，但必须加入日期专属的月标记和日标记，避免普通日期因抽到相同 mood/plan 而重名。

2026-07-16 新增穗状/簇状花材门禁：用户指出当天默认花束中的蛇鞭菊、风信子和金鱼草让整体显著变丑。问题不是单一颜色，而是这些串串花茎太粗、长度和含花量一致、飞到屏幕外，并且蛇鞭菊用外炸花蕊制造了廉价假复杂度。后续金鱼草、风信子、蛇鞭菊等暂缓并入的穗状/簇状具体花材不能只映射成同一个 `SpikeFlower` 后大比例进入每日花束；进入时必须低比例、可回炉，并保证每支有不同长度、开放段、花距、粗细和生长路径。蛇鞭菊应贴轴形成细长绒毛瓶刷感，而不是烟花式爆开。满天星和空气白点可以提高密度，但必须保持轻、小、分散，不把画面重新填满。

### 2026-07-12 审美纠正的长期含义

用户明确要求把本次讨论作为对 Codex 审美判断的重要纠正写入项目，而不是只保存两束日期花的修复结果。后续生成必须继承以下判断：

- **缤纷不是填满。** 中国当代花艺语境里的缤纷来自小花、果点、枝线、局部主花和空气间隔的点状节奏；把不同颜色密铺成一个完整圆球，即使颜色很多，也仍然是一坨。
- **空气感不是简单减量。** 空气由疏密变化、前后距离、大小差异、遮挡关系和边缘伸展共同构成；只减少总数量但保留中心密铺，同样不成立。
- **随机角度不是自然。** 花材必须先服从共同花瓶、花泥或扎口，保持花头到花茎再到锚点的连续关系；角度和弯曲只能在真实生长结构上变化。
- **物理错误会直接破坏审美。** 半空花头、没有茎的花穗、延长后指向不同隐形花瓶的枝条，不是可以用“抽象”解释的风格，而是需要拒绝的生成错误。
- **每日身份属于审美交付。** 日期 seed 在代码里不同还不够；用户必须能从独立花名、日期印记、flowerPlan 和对应地址确认每天确实生成了不同花束。
- **每日花名不能重名。** 可以使用有主观审美的命名词，但默认日期生成必须带有日期专属命名标记，不能让相邻日期或同 plan 日期出现相同中文主名。
- **花材丰富度必须进入生成器。** Primitive Lab 里花型变多不等于主花束自然变丰富；每日 flowerPlan 需要把具象单花、抽象 primitive、果点、枝线和叶材按角色组合起来，不能让新增花型只停留在 LAB。

这些条目是跨日期、跨主题、跨渲染等级的长期门禁。后续不能只针对 `2026-07-07`、`2026-07-11` 或 `2026-07-13` 打补丁，也不能因为某一束通过而删除这段纠正历史。

## 色彩规则

DailyFlora 可以使用高饱和色，但不能让画面变成厚重油画。

色彩成立依赖三件事：

- 花色之间有主次，而不是每个颜色都抢。
- 绿色、叶材和浅色小花负责降温、透气、缓冲。
- 高饱和色应被小花矩阵、果点、线条和局部主花分散承载。

当前较好的方向包括：

- 山岗小花：饱和但开洋。
- 繁星夜色：夜色氛围成立。
- 海盐柠檬：清爽高亮成立。
- 月光手捧：柔和白黄方向成立。
- 晨露莓园空气束：湿润、莓果、晨光和空气感方向成立。
- 夏日风车、梦幻紫、洋水仙季节、会呼吸的风景、狐尾百合、浆果森林和秋日果汁：作为 0.14 当前通过的参考方向保留；后续根据实际页面反馈做局部修正。

2026-07-02 新增并通过的参考：用户提供 `Lychee Garden｜捏一个彩虹🌈` 作为今日审美参考。公开可读信号包括自然风花束、绚丽色彩、彩虹花束、多巴胺配色和“生活是彩色的”。它应作为多色方向的增量证据：允许彩虹式多色，但必须按花材角色分配颜色，并用绿色、浅色花和空气间隔缓冲；果点或荔枝感只能作为有枝条归属的小圆点，不可变成随机散点。此项已进入 `data/aesthetic-review-dashboard.json` 的 `荔枝花园彩虹 / Lychee Garden Rainbow` 复盘卡。`0.13.4` 补上独立 `lychee-garden-rainbow` 地址，`0.13.5` 记录用户确认“做得很好”，状态改为 pass；新花不能借用其他花束地址，若后续被否定，应弃用该地址本身。

## 审美复盘流程

任何主视觉改动前，默认走这个流程：

1. 主动打开审美复盘 Dashboard，读通过基线、可复用规则、日期回炉记录与反向样本；不要等用户提醒整体形态问题后才回看。
2. 读参考图和用户文字，不要只读代码。
3. 把参考图转成人话视觉结论。
4. 写出正向信号和负向约束。
5. 判断需要 primitive、composition、palette、camera/UI 哪一层改变。
6. 若 primitive 不够，先修 Primitive Lab；若 primitive 已够，再改整束 composition。
7. 更新 `data/aesthetic-review-dashboard.json` 和 `docs/aesthetic-review-dashboard.html` 的可读状态。
8. 构建并检查页面，避免 dashboard、花库、花束库入口丢失。
9. 用户验收前，不把 Codex 自己的判断写成 pass。

## 角色分工

审查角色不是人格表演，也不是固定不能变的组织架构。它们是为了把问题拆清楚：

- 创意总监：判断整体方向是否跑偏，守住 DailyFlora 的语境。
- 美术指导：拆色彩、轮廓、花材层级、空气感、前后关系、叶材合理性。
- 项目主任：判断当前阶段是否允许进入下一层，避免绕过门禁。
- 生成架构审查：判断 Three.js 生成结构、参数暴露、比例控制、实例化性能和调试数据是否足够支撑审美目标；它不替代美术判断。
- 花材库管理员：检查 flowerPlan、renderer、Primitive Lab 和 TARGET SHAPE VOCABULARY 是否一致；代码里用到的花型必须登记，参考图里反复出现的形态不能遗漏。

另有一个显式门禁：用户是当前审美验收人。任务不重时，验收就是用户本人。

## 0.13 以后怎么更新本记忆

当用户给出新的关键审美判断时，必须至少更新：

- `CHANGELOG.md`
- `docs/codex-aesthetic-handoff.md`
- `docs/dailyflora-aesthetic-system-0.13.md`
- `data/aesthetic-review-dashboard.json`。新增审美参考必须进入 dashboard，即使还没有图片或还未确认，也要建 `needs-owner-review` 卡片；不能只写入 inspiration library。
- 新增生成花束必须有独立 theme/flowerPlan/URL。不要借用别的花束地址展示新花；如果用户不要这束花，就弃用这个地址本身。

不要把重要判断只留在聊天里。

如果某天用户不给新的审美输入，DailyFlora 仍按本 0.13 审美系统、已确认的花库门禁、既有主题和日期 seed 每日生成；新的外部参考只作为增量校准，不是每天生成的必要条件。

2026-07-02 起，项目有每日自动生成发布任务：每天本机闲时以接近满上下文的独立 Codex cron 执行构建与 GitHub 发布。自动化不能绕过审美审核、owner-provided 参考门禁或新花独立地址规则。
## 2026-07-23 叶片—花型关系定稿

- 用户确认 Strap 与 Palmate 两个叶片原型可以进入受控整合。
- Strap D2 基生原型映射洋水仙、风信子与狐尾百合；Palmate Major Structure Envelope 映射飞燕草。
- 其余偏写实花型继续保持 `unresolved + none`，不使用通用椭圆叶补位。
- 叶片必须属于具体花茎，同一花茎只允许一个 profile 和一种叶序。
- 主花束只消费当前花材计划中实际出现的确认成员；不为叶片修改花朵计划、花朵 RNG 或主要构图。
- 受控接入仍遵守“叶材托住空气、不挡主花、不抢视觉主体”的既有审美规则。

完整矩阵见 `docs/reference-flower-shapes/realistic-leaf-member-mapping-v1.md`。
