一句话总结

开发组

先做目录入口

需要一个开发中的页面总目录：能看到所有条目、版本、debug 入口，再进入每个详细页面。

平台管理

先混放，后剥离

管理花花库和注册用户暂时放在开发组入口里，但将来应该拆成独立后台。

客户页面

补齐公开导航

客户侧需要首页、about、Bouquet Shop、member、downloads，以及特别花束入口的清晰关系。

建议路由结构

/docs/dev-index.html

开发组总目录：列出开发中页面、客户页面 debug 版、版本列表、平台管理临时入口。

/docs/reviews/

审美复盘版本列表；每个版本进入对应 dashboard 详情。

/docs/primitives/

花材库版本列表；每个版本进入对应 Primitive Lab。

/docs/flower-plans/

固定样例列表；默认截图卡片，支持每页 20/50、分类、排序、进入实时详情。

/member 或 /docs/member-test.html

用户页面测试：资料、头像、密码、生成历史、上传记录、积分、订阅、订单、获奖返还。

/admin 或 /docs/admin-test.html

平台管理临时页：花花库、用户、订阅、积分操作日志；将来应从开发组剥离。

缺漏检查

缺入口策略

哪些页面公开，哪些只给开发组

文档说 debug 版都要显示在开发组页面，但没有定义这些入口是否能被公开访问。当前项目是静态站，/docs/ 页面通常可直接访问。

缺账号系统边界

用户、订单、充值、积分需要后端

member 和管理者页面涉及登录、密码、订单、订阅、积分、操作日志，目前文档没有说明数据源、权限、支付、审计存储。

缺版本模型

“版本”还没有统一定义

审美复盘、Primitive Lab、参考图库、固定样例都要求最新版本优先，但文档没有说明版本号、发布日期、变更摘要、状态字段。

缺生成记录规则

复制再生成与不允许热修改

文档说用户能复制并重新生成新一份，同时“不允许热修改”。需要明确：旧记录不可编辑，新记录通过复制参数创建。

缺获奖结算规则

返还积分需要定义触发条件

“生成一次花花的积分乘以 10 返还”需要定义基准积分、评选状态、返还时间、是否可撤销、是否允许人工调整。

缺移动端和加载策略

样例页性能要求需要落地

文档提出截图、分页、排序和分类，但没有确定默认方案。建议默认截图列表，点击后才加载实时渲染。

互联网参考后的补充要求

模块 可借鉴能力 落到 DailyFlora
充值、订阅、订单 Stripe Customer Portal 支持用户管理付款方式、订阅、账单资料和历史发票。 member 页先不要自己造完整支付后台；原型里应拆成“积分余额、订阅状态、订单/发票、充值入口、取消/变更订阅入口”。
密码和账号安全 OWASP 和 NIST 都强调密码重置、认证流程和 MFA 不能成为账号薄弱点。 “修改密码”不应只是表单，需要补：当前密码验证、重置邮件/验证码、登录设备/会话、异常提示不泄露账号是否存在。
平台管理操作 OWASP 对高价值交易建议保留带完整性的审计轨迹，避免被篡改或删除。 手动开订阅、增减积分、改用户资料、标记获奖，都必须记录操作员、时间、IP、变更前后值、原因。
固定样例页性能 MDN 说明 iframe 支持 lazy loading，延迟加载能减少首屏阻塞；但截图列表更适合大量预览。 短期可给 iframe 加 loading="lazy"；中期改成截图卡片，点击后才打开实时渲染详情。
后台权限 OWASP Authorization 指南强调授权逻辑要和业务风险匹配，不能只依赖页面隐藏。 开发组目录可以列入口，但涉及手机号、电邮、积分、订单的管理页不能作为公开静态 docs 暴露。

冲突或风险

  • 平台管理页面“先跟开发组放在一起”和“查看用户手机号/电邮”有权限风险。即使是临时页，也需要最少访问控制，不能和公开 docs 页面同等暴露。
  • 所有客户页面 debug 版集中展示很方便开发，但如果部署到公开站，会暴露未完成页面、测试数据或内部入口。
  • /docs/avatar-lab.html 说“用完之后就删掉”，但它当前已经是构建入口之一。删除前要同步 vite.config.ts 和任何导航链接。
  • 文档末尾的 /special0629-v4/ 路由图片段不完整，像是 Mermaid 图粘贴中断，需要补全后才能作为路由依据。
  • 客户侧 member 和开发组里的用户页面测试功能高度重叠，需要决定一个源页面，避免做两套。

建议优先级

优先级 事项 原因
P0 新增开发组总目录 这是后续所有页面版本、debug 入口和管理临时页的入口骨架。
P0 把固定样例页改为轻量列表 当前多个实时 iframe 会影响滚动和加载，文档明确指出这是问题。
P1 定义统一版本数据结构 审美复盘、花材库、参考图库、样例页都依赖“最新版本在前”。
P1 把 Avatar Lab 改造成用户页面测试页 用户体系是后续积分、订阅、生成记录和获奖返还的核心。
P2 继续补齐其余客户公开页面 about 与 signup 已升级；Bouquet Shop、downloads 仍可按相同品牌系统继续完善。

参考资料

  • Stripe Customer Portal documentation：用于理解订阅、付款方式、账单资料和发票管理可以如何组织。
  • OWASP Authentication Cheat Sheet：用于补充登录、密码、账号安全缺口。
  • OWASP password reset testing guide：用于检查修改密码和找回密码流程。
  • OWASP Logging Cheat Sheet 与 OWASP A09 Logging and Monitoring Failures：用于定义后台人工操作日志。
  • OWASP Authorization Cheat Sheet：用于判断开发组、客户页、平台管理页的权限边界。
  • MDN iframe loading property 与 MDN lazy loading guide：用于固定样例页的加载优化。
